在AI领域，Harness指一种全新的工程范式，它的目标是**解决AI智能体（Agent）在执行复杂、长周期任务时，因“不确定性”而导致的“失控”问题**

以下是构建这套 Harness 的 **5 大核心架构设计模式与工程准则**：

## 1. 极简的核心循环与高度解耦的工具层

- **The Loop is Sacred (神圣的循环)**：Agent 的核心永远是一个极其简单的 `while True` 循环。无论后续添加多么复杂的功能，主循环的逻辑几乎不变，仅仅是“喂上下文 -> 思考 -> 执行 -> 收集结果”的闭环。
- **基于字典的路由分发 (Dispatch Map)**：所有的能力扩展（执行 Bash、读写文件、操作任务、加载知识、甚至唤醒队友）统统抽象为 Tool（工具），并通过一个统一的字典（`TOOL_HANDLERS`）进行解耦路由。这保证了系统极佳的扩展性。

## 2. 上下文保卫战 (Context Window Defense)

大模型的上下文不仅昂贵，而且一旦过载会导致严重的“注意力衰减”和“幻觉”。Harness 必须在“护城河”上投入大量精力：

- **空间隔离 (Subagents)**：把大任务扔给子 Agent 在干净的沙盒里做，主 Agent 只回收“摘要文本”，实现降维打击。
- **按需加载 (Skill Loading)**：拒绝冗长的系统提示词。平时只挂一个“技能目录”，模型真正需要时才通过工具将长篇领域知识注入上下文。
- **三层分级压缩 (Context Compact)**：引入微压缩（清空旧工具输出）、自动压缩（大模型自我总结并重置历史）和手动压缩，保证 Agent 拥有**无限续航**能力，并通过重注入（Re-injection）防止压缩导致的“失忆”。

## 3. 从内存状态到“磁盘即数据库” (Disk-backed Persistence)

真实的软件工程不能依赖脆弱的内存。Harness 必须全面拥抱**文件系统**作为唯一的真实数据源（SSOT）：

- **持久化任务图 (DAG)**：将长线计划拆解为存在依赖关系的 `.tasks/task_n.json` 文件。即便 Agent 重启，工程进度依然存在。
- **文件信箱通信 (JSONL Inbox)**：多 Agent 之间不使用复杂的内存队列或 RPC，而是极其巧妙地采用 Append-only 的 `.jsonl` 文件进行异步通信。这种解耦方式天然防丢、易审计。

## 4. 异步并发与物理隔离 (Concurrency & Isolation)

当 Agent 团队开始成型，必须解决 I/O 阻塞和资源抢占的物理问题：

- **非阻塞执行**：将耗时的命令（如 `npm install`）丢进后台守护线程，主线程通过“信箱排空（Drain Queue）”机制，在下一轮对话前将异步结果强行注入上下文，让 Agent 能够“边等边想”。
- **工作区隔离 (Git Worktree)**：彻底解决多 Agent 并发写代码的冲突问题。将每个任务绑定到一个独立的物理 Git Worktree，Agent 在各自的平行宇宙中执行，完工后再合并销毁。

## 5. 引入规矩：状态机与协议门控 (FSM & Protocols)

自由散漫的 LLM 无法完成严谨的工程，Harness 必须充当“紧箍咒”：

- **单线程聚焦与追责**：强制限制同一时间只能有一个任务处于 `in_progress`，并在模型跑偏时强制注入 `<reminder>` 提醒它回归计划。
- **结构化握手协议**：用带有全局唯一 `request_id` 的 Request-Response 状态机（FSM）来管理高危操作。关机必须申请，重构必须先提交 Plan 审批。
- **从推到拉的自治 (Push to Pull)**：放弃保姆式的任务分配。Agent 具有独立的 WORK/IDLE 生命周期，空闲时会自动轮询任务看板并“抢单”干活。
***
## 总结

这套 Harness 的精髓在于：**不试图让大模型变得全知全能，而是用经典的软件工程手段（沙箱、状态机、队列、文件锁、事件流）在模型外部搭建一个坚固的脚手架。** 大模型只负责做纯粹的“逻辑推理节点”，而 Harness 负责管理时间、空间、记忆和协作规矩。

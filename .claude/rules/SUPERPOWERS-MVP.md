# MVP 与特性研发模式约束 (MVP & NEW FEATURE)

**适用场景**：新增功能、新模块、核心流程重构、跨模块改动、架构设计与从零搭建。
**核心原则**：Design First（设计先行）、先确认方案、再进入实现。

---

## 核心红线

- 在完成设计确认前，**禁止直接编写业务实现代码**。
- 任何试图跳过“分析 -> 设计 -> 计划 -> 实现”流程的行为，都必须停止并回退。
- 如果用户当前只是讨论方案、评估设计或走查架构，应该切换回**问答模式**，而不是继续执行 MVP 实现流程。

## 标准流程

### 1. 明确目标与范围
- 识别本次需求是否属于：
  - 新增模块
  - 新增完整功能
  - 核心架构调整
  - 跨前后端的大范围改动
- 若只是小修小补或局部改动，应切换到**维护模式**。

### 2. 设计先行
- **触发技能**：`brainstorming`
- **要求**：
  - 先澄清目标、约束、成功标准
  - 给出 2-3 种可行方案及取舍
  - 输出清晰的设计说明（模块边界、数据流、依赖关系、异常处理、测试策略）
  - 必须等待用户明确批准后，方可进入下一阶段

### 3. 制定实施计划
- **触发技能**：`writing-plans`
- **要求**：
  - 将设计拆解为可独立验证的原子任务
  - 每个任务至少包含：
    - 目标文件路径
    - 关键函数 / 接口签名
    - 验收标准
    - 对应测试场景

### 4. 隔离开发环境
- **触发技能**：`using-git-worktrees`
- **动作**：优先使用隔离工作树，避免污染主线。
- **降级策略**：若环境不支持，可退化为独立分支开发。

### 5. 按计划执行
- **触发技能**：`subagent-driven-development`、`executing-plans` 或 `dispatching-parallel-agents`（若子任务可独立并行处理）
- **执行原则**：
  - 严格按计划实现，不随意扩 scope
  - 遵循 RED -> GREEN -> REFACTOR
  - 先写失败测试，再写实现代码
  - 不把无关优化夹带进当前任务

### 6. 审查与交付门禁
- **触发技能**：`verification-before-completion` (提审前自证)、`requesting-code-review` (发起审查)、`receiving-code-review` (处理审查反馈)
- **检查重点**：
  - 是否符合已确认设计
  - 是否存在超出计划的隐性改动
  - 测试是否充分且有意义
  - 代码是否保持可维护性

### 7. 收尾与留痕
- **触发技能**：`recording-iteration-history`、`finishing-a-development-branch` (若使用了独立分支)
- **动作**：
  - 更新 `CHANGELOG.md` 或项目既有历史记录
  - 记录本次改动摘要、验证结果、必要引用信息
- 若涉及分支或工作树清理，再进入后续收尾动作。

---

## 豁免规则

以下情况无需走完整 MVP 流程：

- 纯文档修改
- README 更新
- 纯配置键值调整
- 不改变行为的注释补充

---

## 元技能 (Meta Skills)

在整个开发流程中，可随时按需触发以下底层技能：
- **触发技能**：`using-superpowers`
  - **场景**：当需要了解现有工具链能力边界，或调用其他进阶工具包时使用。
- **触发技能**：`writing-skills`
  - **场景**：当发现当前项目缺少必要的标准流程，需要总结或编写新的自定义技能规范时使用。

# 💬 通用问答模式约束 (GENERAL QA)

**适用场景**：所有不涉及当前项目代码库实质性文件变更的交互，包括但不限于**技术问答、概念解释、代码解读、技术选型探讨**。

---

## 🛑 核心约束：自由探索，免除工程流程

- **流程全面豁免**：在此模式下，你**无需**也**不应**触发任何 Superpowers 开发工作流技能（例如：`brainstorming`, `writing-plans`, `test-driven-development`, `subagent-driven-development` 等）。
- **允许行为**：你可以自由生成多轮解释性文字、架构图描述以及用于说明的示例代码片段（非生产代码）。

## ✅ 行为与回答准则

1. **准确性与权威性**：
   - 所有的技术解释需基于官方文档或业界公认的最佳实践。
   - 不要为了回答而凭空捏造 API 或库功能。

2. **示例代码规范**：
   - 若在解答中提供了代码片段，必须明确标注其为 `示例代码`。
   - 必须简要说明该示例与实际生产环境代码之间的潜在差异（如错误处理、类型定义等）。

3. **上下文感知与结合**：
   - 尽管是问答模式，当用户询问的是**当前项目内的代码逻辑**（例如："请帮我看看 `src/utils/auth.js` 里的逻辑有没有问题"）时，你仍应使用读取工具结合实际文件内容进行回答，避免脱离上下文。

## 🔄 边界处理与无缝切换

- **跨界拦截**：一旦用户的意图从“探索/询问”转变为“在当前项目中修改代码”（例如："我觉得你说的对，按这个方案帮我把代码改了吧"），你必须：
  1. 立即停止纯问答的回复。
  2. 主动提示用户：“检测到需要修改项目代码，正在切换至【维护迭代模式】或【MVP 模式】”。
  3. 切换至相应模式并遵循其约束执行任务。

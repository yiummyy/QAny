## 1. 核心工作流与模式路由 (CRITICAL)

本项目以 `CLAUDE.md` 与 `.claude/rules/*` 作为主规则来源。在开始处理任何用户请求前，你**必须首先**进行意图识别与模式路由：

- **强制第一步**：读取并应用 `.claude/rules/MODE_ROUTER.md` 的路由逻辑。
- **自动分流**：根据用户指令，在首次回复开头声明并激活以下三种模式之一：
  1. **MVP 模式**：@`.claude/rules/SUPERPOWERS-MVP.md`
  2. **维护模式**：@`.claude/rules/SUPERPOWERS-MAINTENANCE.md`
  3. **问答模式**：@`.claude/rules/GENERAL_QA.md`
- **Skills 使用原则**：
  - `skills` 仅作为补充能力库，在任务明确相关时再使用。
  - `SessionStart` 注入内容只是轻量提醒，不是本项目的主流程来源。
  - 若 `skills` 内容与本文件或模式规则冲突，以本文件和已选模式规则为准。
- **模式优先级说明**：
  - 路由判断以 `MODE_ROUTER.md` 为唯一入口。
  - 一旦完成模式判定，后续只遵循对应模式文件，不再额外叠加其他流程规则。
  - 若用户意图发生变化，必须先声明模式切换，再按新模式继续处理。

***

## 2. 编码规范与偏好

- **沟通语言**：与用户沟通时，默认使用**中文**。
- **代码风格**：
  - 变量和函数命名应具有高度自解释性（Self-documenting）。
  - 避免过早优化（YAGNI），优先保证代码可读性与可维护性。
  - 尽量保持实现简单直接，避免无必要抽象。
- **测试要求**：
  - 新增核心逻辑时，优先补充针对性的自动化测试。
  - 若涉及 Bug 修复，优先编写可复现问题的测试。
- **Git 提交规范**：遵循 Conventional Commits 格式，如 `feat:`、`fix:`、`docs:`、`refactor:`。

## 3. 项目常用命令速查

### 前端（项目根目录）

- **启动开发服务器**：`npm run dev`
- **构建生产版本**：`npm run build`
- **运行代码检查**：`npm run lint`

### 后端（`backend` 目录）

- **运行测试**：`pytest tests/ -v`
- **运行代码检查**：`ruff check .`

### Docker（项目根目录）

- **启动服务**：`./deploy.sh start`
- **停止服务**：`./deploy.sh stop`
- **查看日志**：`./deploy.sh logs`
- **查看状态**：`./deploy.sh ps`

## 5. 执行原则补充

- 对于**纯分析、走查、评估、代码解读、配置审阅**类请求，优先进入**问答模式**，避免误触发实现流程。
- 对于**文档、注释、说明文字**类小改动，按维护模式中的轻量路径处理。
- 对于**新增功能、架构调整、跨模块变更**，进入 MVP 模式并遵循 Design First。
- 若任务已能由本文件与 `.claude/rules/*` 明确约束，则优先遵循项目规则，不要额外引入无关流程说明。
